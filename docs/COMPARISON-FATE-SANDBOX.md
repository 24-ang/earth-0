# earth-0 与 fate-sandbox 差异性与玩法探索对比日志

> **致敬与鸣谢**：本项目 `earth-0` 的底层规则约束与 Agent 架构思想深深受益于先驱项目 `fate-sandbox`。在此，对 `fate-sandbox` 及其作者在交互式叙事沙盒领域的探索与开源精神致以最诚挚的敬意。本项目的开发是建立在先驱者奠定的技术基石之上的一步微小实践。

---

## 一、 fate-sandbox 在工程纪律与严谨性上表现更出色的地方

作为后来者，`earth-0` 更加追求快速的功能探索与玩法落地，这也导致我们在技术底座的严谨性上与 `fate-sandbox` 存在客观的技术债差距：

### 1. 类型安全与工程纪律（TypeScript 约束层）
*   **前作（fate）**：极其严谨。启用了严格模式（`strict: true`），配合 `noUncheckedIndexedAccess` 和 `noUnusedLocals`，代码库内做到了近乎零 `any`。测试覆盖了细致的领域层与审计层。
*   **本作（earth-0）**：由于功能迭代过快，代码库内遗留了大量 `as any` 类型断言。虽然当前测试（230 cases）能全部通过，但类型安全的防线在逐步收紧中。

### 2. 事务性状态修改（State Transaction Invariant）
*   **前作（fate）**：工具的执行采用了 `clone` → `execute` → `validate` → `commit` 四阶段事务机制。一旦中间出错，状态自动回滚，绝不留脏数据。
*   **本作（earth-0）**：目前工具执行仍是直接修改内存状态对象，依赖于结算结束后的 `saveState()` 进行兜底落盘，暂不具备强事务隔离回滚能力。

### 3. 洁净室渲染（Clean-Room Rendering）
*   **前作（fate）**：采用彻底的双模型物理隔离。逻辑引擎只产出结构化的 JSON Packet（Direction Packet），再单独传给叙事渲染模型。渲染模型物理上接触不到任何游戏内部状态，防泄密与防超游做到了极致。
*   **本作（earth-0）**：2026-06 月采用单模型三段式 Prompt Contract 约束。2026-07-01 升级为**裸 stream 结构性隔离**：Phase 3 渲染 LLM 走 `generateCompletion`（物理上无 tool definitions），NPC 台词由 Phase 2 独立 LLM 生成后原文引用不改写，渲染 LLM 作为"剪辑师"而非"主笔"。相比 fate 的 Packet 路径更松（允许渲染 LLM 看到完整的 NPC 回应文本和空间上下文），但比旧版 prompt 软约束更硬（物理隔离工具列表，不靠 prompt 请求 LLM "不要调工具"）。

## 二、 earth-0 基于“反上帝视角”进行的系统性拓展玩法探索

相比于前作更侧重于型月宇宙下的“主线故事推进（Narrative-driven）”，`earth-0` 试图解决的核心命题是：**在实现“心智与信息完全隔离（反上帝视角）”之后，我们能解锁哪些前所未有的沙盒扩展玩法？** 

为此，我们开发了一系列相互交织的世界模拟系统：

### 1. 社交记忆扩散系统（Emergent Rumor Propagation）
由于每个 NPC 都是信息隔离的，这使得“八卦和秘密在世界中自然传播”成为可能。NPC 在同一个房间碰面时，引擎会基于他们之间的亲密关系度，自动交换对方没有的记忆标签（`memoryTags`），流言会随着 NPC 在地图上的移动而扩散，玩家需要通过互动主动去打听或揭示秘密。

### 2. 实体物理空间与视野系统（Spatial Grid & Perception）
房间不再是抽象的背景版，而是具备网格坐标、墙壁拦截以及家具容器的物理空间。由此衍生出了**目击与感知检定（Perception Check）**：玩家偷窃 NPC 物品或金钱时，引擎会校验同房间内其他 NPC 的相对距离与视线。若被目击，将自动引发关系惩罚并写入目击记忆。

### 3. 自主日程移动与短信投递（Schedules & Messaging Simulation）
NPC 会根据年龄阶段和星期，在地图网格中自主进行位置迁移。当他们发现自己移动到了新地点，或者在某处碰到了其他角色，且与玩家的好感度足够时，他们会从自己的记忆库中抽取相关事实，主动给玩家发送带有情境感的动态短信（“刚才在走廊碰到雪乃了”），使手机与 SNS 系统拥有真实的“活着”的感觉。

### 4. 信息可见性与秘密防火墙（Information Security Level）
世界常识（Lore）、角色事实（Fact）和剧情揭示（Reveal）均采用了严格的分级过滤机制。NPC 不再能够随时抽取全部世界观，而是随着关系好感度阶段（陌生 -> 朋友 -> 闺蜜 -> 恋人）的提升，逐步在 Prompts 中解锁更隐私的事实。

## 三、 故意不同的设计选择

这些差异并不代表优劣，而是源于两个项目核心设计愿景的分野：

| 维度 | fate-sandbox 的选择 | earth-0 的选择 | 差异化考量 |
|------|-----------|---------------|------|
| **状态可见性** | 公开/秘密状态在物理上彻底拆分 | 三套正交的分级过滤系统，引擎级运行时过滤 | fate 题材依赖“真名/宝具”等绝对秘密泄露；earth-0 则采用 D&D 式的“GM 全知，由引擎根据关系动态分发”的哲学。 |
| **世界架构** | 单一的型月 campaign 世界线 | 完全解耦的通用引擎 + 可插拔的 **Worldpack** 数据包 | fate 聚焦于型月世界的高精度还原；earth-0 追求的是“彻底的题材无关性”，任何人都可以通过配置文件无缝换掉整个世界观（如换成 Wasteland 包）。 |
| **角色自发性** | GM 决定 NPC 行为，通过结构化包约束 NPC 言行 | NPC 并行派生独立心智，GM 无法控制 NPC 真实言行 | earth-0 追求 NPC 的“心境自主权”，NPC 可以嘴上妥协、内心抗拒，甚至因信息差产生误判，GM 只负责编织他们冲突后的最终表现。 |

## 四、 纯粹由于开发阶段较早尚未完善的赤字

*   **TypeScript Strict 模式的完全推行**：当前类型债务尚待逐步清理。
*   **存档文件的同盘原子化写入**：未来应引入 `.tmp` 写入加 `renameSync` 原子提交以规避崩溃断电导致存档损坏的隐患。
*   **合并冗余的底层管理工具**：如 `spawn_npc_agent` 与 `spawn_npc_agents` 之间的重复代码提取，精简工具目录。
