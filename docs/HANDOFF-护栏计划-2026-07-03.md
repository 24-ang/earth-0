## 一、根本问题：不是代码烂，是"接缝没有护栏"

这个项目的 bug 有一个统一根因：**模块本身大多没坏，bug 全住在模块与模块之间的缝里。**

具体由三层原因叠加：

### 1.1 类型护栏被关掉了（最致命）
- `engine/state.ts` 是 **4124 行的神对象**，导出 ~134 个符号。
- `tsconfig.json` 里 `noImplicitAny: false`、`noUncheckedIndexedAccess: false`。
- 全库 **281 处 `as any`**，其中 state.ts 独占 78 处——**最核心的数据结构，类型检查基本是关闭的。**
- 后果：模块 A 写进 state 的数据结构，和模块 B 读出来时期望的结构，中间的"约定"**全靠人脑记，编译器不拦**。每次接力就漂移一点。

### 1.2 文件管理混乱（用户亲自点出的，确实是根因）
- **数据脑裂**：`data/` 和 `worldpacks/oregairu/` 有大量同名文件。`data/` 下的是 TS 静态导入的兜底模板，**运行时不生效**；真正生效的是 `worldpacks/`。改错地方 = 静默无效。
- **空兜底文件陷阱**：`data/residence_templates.json` 的内容就是 `{}`。引擎启动时 `loadActiveWorld()` 本应把 worldpack 的真数据覆写进内存变量——**但如果框架没有调用 loadActiveWorld（headless/某些入口），你拿到的就是空 `{}`，且无报错**。本轮的"住宅不展开"bug 就是这么来的。
- **167 个文件**，工具散落在 action/lookup/state/tui 四类，加载顺序和时机没有单一权威入口。

### 1.3 多 LLM 接力 = 制造熵
- 项目被 Claude / Codex / Gemini / DeepSeek 轮流改过。
- 每个模型每次只带**部分上下文**进来 → 看到一个 bug → 局部修复 → 顺手戳破另一条缝 → 下一个模型再修那条 → 再戳破一条。
- **这就是"越改越多"的机制。** 不是谁蠢，是"接力 + 无护栏"必然产生熵。

### 1.4 测试绿 ≠ 游戏能玩（bug 隐身的原因）
- 274 个 `test.ts` 全过，但一开游戏全是 bug。
- 因为它们是**单元测试**——每个模块喂干净输入单独测。**没有一个测试跑完整管线**（init_game → init_profile → 渲染 → 拿最终玩家状态断言完好）。
- 加上大量**静默 `catch`** 和 init_profile 里的**静默回滚**（try 块任一步抛异常 → 整个玩家状态回滚到快照，但只在返回文本里小声说"失败"）——bug 被系统性地藏起来，直到你半夜开游戏才炸。

## 二、本轮（2026-07-03）已经修的

**引擎层**
- `engine/state-grid.ts`
  - `initPlayerGrid()`：未知/室外位置不再返回 `gridPos: null`，改为默认 `[5,5]`（room 存在时仍按 exit/door/floor 优先级找坐标）。
  - `instantiateResidence()`：内存中没有模板时，**直接从 `worldpacks/<world>/residence_templates.json` 兜底读取**（对抗 1.2 的空兜底陷阱）。
- `tools/state/init_game.ts`：加了**最小兜底**——内衣（inner_top/inner_bot）、按龄最低生活费、生殖器档案自动生成（getOrCreateSexState）、world flag。返回**结构化缺口报告**（✅已填充/⚠️部分/❌未填充 + 建议工具 + 可用模板）。
- `tools/state/init_profile.ts`
  - 装备**叠加而非覆盖**（保留 init_game 的兜底内衣，之前 `= {}` 把内衣清掉了）。
  - **角色数据库自动合并**：玩家名匹配 characters.json 里的角色时，自动用原作数据覆盖模板的体型/属性/技能/标签→flags/装备。**这是关键设计——LLM 不用手动补，引擎自动查库。**
  - 单件角色装备校验失败时 **skip 而不是整个回滚**（之前一个缺 volume 的字段就让整个初始化回滚）。
- `engine/phase3-render.ts`：`buildSceneBrief` 显式注入玩家名（防止渲染出现 ghost 名字"维"——那是 createInitialState 的默认名）。
- `tools/lookup/lookup_lore.ts`：TF-IDF 检索结果加**关键词兜底过滤**（之前搜"材木座"返回甘城/喰种/EVA 等完全无关世界包的条目）。

**数据层**
- `worldpacks/oregairu/characters.json`：补了**材木座义辉**的完整条目（此前完全不存在！82kg/偏胖/圆框眼镜/写作Lv5/游戏部/2年C班/中二病标签）。
- `worldpacks/oregairu/init_profiles.json`：修正关系标注（删掉串线的"WLW集团"、把错误的"同班"改为"同校"），给模板加了 skills/contacts/relationships/memories/residence 字段。

**基线：`npx tsx test.ts` = 274 passed, 0 failed。**

## 三、下一步计划：装护栏（这才是止血的关键）

**不要再一个一个手修接缝 bug——那是永动跑步机。** 要做的是把"看不见的 bug"变成"看得见的、立刻报错的 bug"。

### 计划 A（核心，必做）：状态完整性校验器
**新建 `engine/validate-state.ts`**，导出 `validatePlayerState(gs): {ok: boolean, errors: string[], warnings: string[]}`：
- 检查玩家状态的完整性不变量，例如：
  - `gridPos` 不为 null（除非明确的无网格位置）
  - `equipment` 至少有内衣两件，且每件都有 name/type/slot/weight/volume
  - `funds` ≥ 0，`hp.current`/`hp.max` 合法
  - 生殖器档案存在（sexStates[player.name]）
  - 手机 phoneData 已初始化（如果背包里有手机）
  - `location` 能被 getRoom 解析到（不是幽灵地点）
  - flags 里 world flag 存在
- **发现问题就大声 `console.error` + 在返回里列清单**，绝不静默。
- 关键点：**校验器只报告，不修复**——修复是工具的责任，校验器是"烟雾报警器"。

### 计划 B（核心，必做）：端到端冒烟测试
**新建 `e2e-init-test.ts`（或加进现有 e2e-test.ts）**：
- 真跑完整管线：`init_game("材木座义辉","男",16)` → `init_profile("千叶市高中生")` → `validatePlayerState()`。
- 断言最终状态完好：有内衣、gridPos 非 null、有制服、有眼镜、体型=偏胖、住宅"家"展开且有房间、通讯录有联系人、flags 含中二病。
- 再跑一个"数据库无此角色"的场景（如"杀手"）：断言兜底生效 + 缺口报告正确。
- **这个测试就是防止"测试绿但游戏炸"的护栏。**

### 计划 C（做完 A/B 后写进规则）：把护栏变成铁律
- 在 `CLAUDE.md` 三条铁律里加第四条：**"改完初始化/状态相关代码，必须跑 e2e-init-test 且 validatePlayerState 全绿。静默回滚一律改为报错。"**
- 这样以后交给 **任何模型（包括 DS）** 的每一条指令都自带护栏——它一破坏接缝，测试立刻红，不会等你开游戏才发现。**跑步机就停了。**

### 计划 D（可选，低优先，别现在做）
- 消灭静默 catch（CLAUDE.md 已有规范，但库里还有残留）。
- data/ 空兜底文件的脑裂：要么让 loadActiveWorld 成为唯一权威入口并在启动时强制调用，要么把静态导入的空 `{}` 换成会在使用时报错的守卫。
- state.ts 神对象拆分——**这是大工程，没有护栏之前不要碰，会引入新 bug。**

## 四、明确不要做的事

- ❌ **不要**继续"发现一个 bug 修一个"而不装校验器——那是无底洞。
- ❌ **不要**现在就去拆 state.ts 神对象——没护栏时拆它=灾难。
- ❌ **不要**开多 agent 全量审计 workflow——烧钱，且用户额度紧张。校验器+冒烟测试用普通对话就能写完。
- ❌ **不要**在没读 PHILOSOPHY.md / decisions.md 的情况下质疑现有分层设计然后推倒重来。
- ❌ **不要**改 `data/` 下的世界数据（改 `worldpacks/oregairu/`）。
