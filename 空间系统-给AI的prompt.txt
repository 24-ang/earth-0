你是一个游戏系统设计师。请帮我评估一个空间感知模块的设计方向。

## 项目背景

我在做一个终端文字游戏（earth-0），现代日本背景，多作品角色共存。架构是 data/（静态数据）→ engine/（TypeScript 规则计算）→ LLM（叙事）。

**关键约束：**
- 运行在 **Android Termux**（手机终端），不是台式机/服务器
- 运行时是 **pi coding agent**（Node.js CLI 工具），不是自研引擎
- 零编译——TypeScript 由 pi 直接执行，没有 build step
- 内存/CPU/存储都有限——不要复杂数据结构，不要图形渲染，不要独立 NPC agent
- engine 的职责是读写 JSON 文件 + 把函数注册为 pi 工具供 LLM 调用——不做事件循环、不做后台线程
- 有 TUI 命令系统（/status /look /inventory 等）——玩家可直接查看面板，不经过 LLM

目前已有系统：骰子、战斗、HP/AC、负重、物品偷窃/装备、Layer1 性欲模块、session 持久化。

现在要做两个新功能：**空间感知** 和 **模拟人生式建造**。

## 当前困境

我找了一个 AI（Claude）帮我设计的，但它第一版方案被我自己否了——它想把空间感「靠 LLM 描写」来实现。我反驳的要点：

1. LLM 不可能每轮描写房间——玩家在房间里走两步就撞墙了，空间感来自**行为受限**，不是来自文字
2. 教室比操场小这件事，不该靠每次提一嘴「这里很挤」，而是靠**操场能跑、教室不能跑**——行为差异自然传递空间感
3. 教室放 30 套课桌椅不会「转身都困难」——那个 AI 的描写脱离常识

所以我把方向纠正为：空间 = 约束系统。房间有坐标和尺寸，家具有位置和阻挡属性。引擎算移动判定，LLM 只收结论叙事。

## 目前的设计草案

### 数据层
```json
"2年J班": {
  "size": [8, 6],              // 米，宽×深
  "furniture": [
    { "name": "课桌", "pos": [1, 0], "size": [0.6, 0.4], "block": true }
  ],
  "exits": { "南": "走廊" }
}
```
操场就是 `"size": [100, 60]`。

### 引擎做的事
- 玩家「往前走」→ 引擎算前方 2m 有课桌阻挡 → 返回「你走到课桌前，没法再往前」
- 玩家「跑到对面」→ 操场 100m 需 12 秒，教室 8m 需 1 秒
- 玩家「找地方躲」→ 引擎查家具 block 属性
- `/room` 命令 → TUI 面板显示当前位置、尺寸、家具、出口

### 建造
- 玩家买/拆/挪家具 → 引擎更新坐标 → 持久化到 JSON。模拟人生那套。

## 我的问题

1. **这个方向对吗？** 坐标系统对文字游戏来说是不是太重了？有没有更轻的方案能达到同样的「空间 = 约束」效果？

2. **坐标粒度怎么定？** 每个房间真的需要 [x,y] 坐标吗，还是用更抽象的「区域划分」就够了？（比如教室分成「前排」「后排」「窗边」「门口」）

3. **和 LLM 的交互协议怎么设计？** 引擎算完空间限制后，传给 LLM 的格式是什么？是纯文本描述，还是结构化数据？

4. **建造系统的边界在哪？** 玩家能在别人教室里盖房子吗？能在街上放家具吗？需要所有权/领地概念吗？

5. **如果不用坐标，有什么替代方案？** 我想到的只有两种：(a) 精确坐标碰撞、(b) 模糊区域+距离描述。有没有第三种？

请用中文回答。不需要代码，帮我理清设计方向就行。
